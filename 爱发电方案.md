## 🟢 第一部分：业务架构要求 (Business Requirements)

### 1. 业务流程图
由于缺乏国内企业资质，本系统采用「爱发电 (Aifadian)」作为代收款平台。
1. **前端生成订单**：前端拼接带有当前用户内部ID（`custom_order_id`）的爱发电付款链接。
2. **跳转支付**：用户跳转至爱发电完成微信/支付宝付款。
3. **Webhook 被动发货 (主流程)**：爱发电向我们的后端发送 POST 请求，后端校验 RSA 签名后，解析 `custom_order_id` 为目标用户增加账户算力/时长。
4. **API 主动查询 (兜底流程)**：若 Webhook 漏单，提供主动查询最新订单的 Service 工具类，利用 MD5 签名请求爱发电 API，比对数据库进行补单。

### 2. 数据库设计建议 (Order Table)
请为我设计一个订单表（推荐使用 Prisma / TypeORM 或原生 SQL Schema），必须包含以下字段：
* `id`: 内部流水号 (UUID / Auto Increment)
* `out_trade_no`: 爱发电的全局唯一订单号 (String, Unique Index)
* `system_user_id`: 对应爱发电回调中的 `custom_order_id` (String)
* `plan_id`: 用户购买的套餐ID (String)
* `amount`: 实际付款金额 (Decimal)
* `status`: 订单状态 (Enum: `PENDING`, `COMPLETED`)
* `created_at`, `updated_at`

### 3. 环境变量清单 (.env)
请在代码中引用以下环境变量：
* `AFDIAN_USER_ID`: 爱发电创作者 UID
* `AFDIAN_API_TOKEN`: 用于主动请求 API 的 Token
* `AFDIAN_PUBLIC_KEY`: 用于 Webhook 验签的 RSA 公钥（包含 `-----BEGIN PUBLIC KEY-----` 头尾）

### 二、 API 说明 (主动查询)
当前功能需要用到 user_id 和 生成的API Token，以下简称 token。请求平台时可以为form表单或者json，平台返回的数据为json。

#### 1. API 签名介绍 (MD5)
为了保证数据安全与灵活性，平台做了一个简单的签名逻辑。平台接收的参数如下：
* `user_id`: 上面的 user_id，表明你是谁
* `params`: 具体接口传参的json字符串
* `ts`: 发出请求时秒级时间戳
* `sign`: 针对上面3个数据的签名，防止伪造数据。

sign 的计算规则为：`md5(token+请求数据按key排序拼接key和value)`
具体到这次场景，参数key为固定值，只有上面四个字段 sign 签名可以简化为：
`sign = md5({token}params{params}ts{ts}user_id{user_id})`
注意：{}包的数据为具体数值，token直接写具体值，其它参数写kv，没有任何连接字符，直接拼接。

示例：
假设当前数据如下
user_id: abc
params: {"a":333}
ts: 1624339905
token: 123 (注意这个不传递到服务端，仅参与签名计算)

计算过程：
sign = md5('123params{"a":333}ts1624339905user_idabc')
得出 sign = a4acc28b81598b7e5d84ebdc3e91710c

#### 2. JSON请求示例
