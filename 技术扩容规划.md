> 基于当前架构（FastAPI + SQLite + 单进程 uvicorn + 腾讯云 COS）的分阶段扩容方案。

## 当前架构基线

| 组件 | 现状 | 能承载并发 |
|------|------|-----------|
| 数据库 | ~~SQLite（单文件，写串行）~~ **PostgreSQL（Supabase）✅ 已迁移** | ~5000 并发写 |
| Web 服务 | uvicorn 单进程 | ~100 并发请求 |
| TTS | ~~同步阻塞（每请求占线程 5~30s）~~ **线程池异步化 ✅ 已修复** | ~30 并发 TTS |


```python
# backend/app/worker.py
async def tts_task(ctx, text: str, reference_id: str, message_id: int, user_id: int):
```

```python
# chat.py TTS 接口改造
@router.post("/tts/submit")
async def submit_tts(data: TTSRequest, ...):
    task = await arq_pool.enqueue_job('tts_task', data.text, voice_ref, data.message_id, user_id)

@router.get("/tts/status/{task_id}")

```

**前端改造（参照 TTI 轮询逻辑）：**

```typescript
startTTS: (messageId, text, characterId, voiceRef) => {
  chatApi.submitTTS(text, characterId, voiceRef, messageId).then(res => {
    if (res.task_id) {
      const poll = () => {
        chatApi.checkTTSStatus(res.task_id).then(status => {
          if (status.audio_url) {
            get().setAudioCache(messageId, status.audio_url)
            get().playAudio(messageId, status.audio_url)
          } else {
            setTimeout(poll, 2000)
          }
        })
      }
      poll()
    }
  })
}
```

**预期收益：** TTS 从线程池变任务队列，峰值能力取决于 worker 数量，彻底不阻塞 API 层。

#### 3. 引入 Redis

- ARQ 任务队列存储（TTS 队列必须）
- 积分余额缓存（减少数据库查询）

```bash
    - redis_data:/data
```

#### 4. 限流改为 Redis 共享（多进程一致）

当前 slowapi 使用内存存储，多 worker 下限流不准确。

```python

    key_func=get_remote_address,
    storage_uri="redis://localhost:6379"
```

单台服务器的 CPU、内存、网络带宽已成瓶颈，需要水平扩展（多台机器）。同时 AI 对话（LLM 调用）的延迟和成本成为主要矛盾。

#### 1. 容器化 + 水平扩展

**改动：** `docker-compose.yml` → Kubernetes 或 Docker Swarm


#### 2. 数据库读写分离

Supabase 支持只读副本（Read Replicas），开启后：

```
写操作（发消息、扣积分）→ 主库
读操作（加载历史、查角色）→ 只读副本（1~3个）
```

```python
# database.py 配置读写分离
write_engine = create_engine(settings.DATABASE_WRITE_URL, ...)
read_engine  = create_engine(settings.DATABASE_READ_URL, ...)
```

聊天历史查询、角色列表等读请求约占总请求 70%，从库分担后主库压力大幅下降。

#### 3. CDN + 边缘缓存

- 所有媒体资源（TTS 音频、TTI 图片、角色头像）强制走腾讯云 CDN
- 前端静态资源（JS/CSS）部署到 CDN，与后端服务分离
- Next.js 应用迁移到 Vercel 或腾讯云 EdgeOne

#### 4. LLM 调用优化

```python
# 确认 SSE 流式返回已稳定

if len(history) > 20:
```

**请求去重缓存：** 当前已实现 `md5(text+voice_id)` 缓存 key，已是最优策略。

```
用户发消息 → API → 写入消息队列 → AI Worker 消费 → 流式推送给用户
```

对 AI 回复、TTS、TTI 全部异步化，API 层只负责接收和推送，计算全部在 Worker。

```python
    def set_sqlite_pragma(dbapi_conn, _):
        cursor = dbapi_conn.cursor()
        cursor.execute("PRAGMA journal_mode=WAL")
```

#### 3. 引入 Redis

**用途：**
- ARQ 任务队列存储（TTS 队列必须）
- 积分余额缓存（减少数据库查询）
- 会话限流计数器（当前 slowapi 内存限流多进程不共享）

```bash
# docker-compose.yml 新增 Redis 服务
redis:
  image: redis:7-alpine
  ports:
    - "6379:6379"
  volumes:
    - redis_data:/data
```

```python
# 积分查询缓存（可选）
async def get_user_credits(user_id: int) -> int:
    key = f"credits:{user_id}"
    cached = await redis.get(key)
    if cached:
        return int(cached)
    credits = await db_query_credits(user_id)
    await redis.setex(key, 60, credits)  # 缓存 60 秒
    return credits
```

#### 4. 限流改为 Redis 共享（多进程一致）

当前 slowapi 使用内存存储，多 worker 下限流不准确。

```python
# backend/app/main.py
from slowapi.util import get_remote_address
from redis import Redis

# 改为 Redis 后端
limiter = Limiter(
    key_func=get_remote_address,
    storage_uri="redis://localhost:6379"
)
```

## 阶段三：支撑 100000 注册用户

> 预估高峰并发：3000~8000  
> 改造周期：1~3 个月  
> 改造难度：★★★★☆

### 核心问题

单台服务器的 CPU、内存、网络带宽已成瓶颈，需要水平扩展（多台机器）。同时 AI 对话（LLM 调用）的延迟和成本成为主要矛盾。

---

### 必做项

#### 1. 容器化 + 水平扩展

**改动：** `docker-compose.yml` → Kubernetes 或 Docker Swarm

```yaml
# docker-compose.yml 改为支持副本扩展
services:
  backend:
    image: nyx-ai-backend
    deploy:
      replicas: 4          # 4 个后端实例
      resources:
        limits:
          cpus: '2'
          memory: 2G
    
  tts-worker:
    image: nyx-ai-backend
    command: arq app.worker.WorkerSettings
    deploy:
      replicas: 8          # 8 个 TTS worker，并行处理语音任务
```

---

#### 2. 数据库读写分离

PostgreSQL 主从架构：主库写，从库读。

```
写操作（发消息、扣积分）→ 主库
读操作（加载历史、查角色）→ 从库（1~3个）
```

```python
# database.py 配置读写分离
write_engine = create_async_engine(settings.DATABASE_WRITE_URL)
read_engine  = create_async_engine(settings.DATABASE_READ_URL)
```

聊天历史查询、角色列表等读请求约占总请求 70%，从库分担后主库压力大幅下降。

---

#### 3. CDN + 边缘缓存

- 所有媒体资源（TTS 音频、TTI 图片、角色头像）强制走腾讯云 CDN
- 前端静态资源（JS/CSS）部署到 CDN，与后端服务分离
- Next.js 应用迁移到 Vercel 或腾讯云 EdgeOne

```
用户（北京）→ CDN 北京节点 → 命中缓存直接返回（< 10ms）
用户（北京）→ CDN 北京节点 → 回源腾讯云 COS（首次，< 50ms）
```

#### 4. LLM 调用优化

AI 对话是成本和延迟的主要来源：

**流式输出（已有基础，完善前端展示）：**
```python
# 确认 SSE 流式返回已稳定
async def stream_chat_response(text: str):
    async for chunk in llm_service.stream(text):
        yield f"data: {chunk}\n\n"
```

**对话上下文压缩：**
```python
# 超过 20 条消息时，压缩旧消息为摘要，减少 token 消耗
if len(history) > 20:
    summary = await compress_history(history[:-10])
    history = [summary_message] + history[-10:]
```

**请求去重缓存（相同文本 TTS 缓存复用）：**
当前已实现 `md5(text+voice_id)` 缓存 key，已是最优策略。

---

#### 5. 消息队列解耦（RabbitMQ / Kafka）

当并发极高时，直接调用服务会产生压力传导。引入消息队列作为缓冲：

```
用户发消息 → API → 写入消息队列 → AI Worker 消费 → 流式推送给用户
```

对 AI 回复、TTS、TTI 全部异步化，API 层只负责接收和推送，计算全部在 Worker。
